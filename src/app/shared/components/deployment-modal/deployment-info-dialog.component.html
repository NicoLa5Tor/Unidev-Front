Cómo preparar el repositorio

Antes de desplegar

Buscamos archivos en este orden, siempre en la raíz del repositorio (al mismo nivel que el archivo README.md):

  1. Primero buscamos docker-compose.yml en la raíz.
    Use esta opción si su aplicación es un stack completo (por ejemplo: backend + frontend + base de datos, o varios microservicios). Al menos uno de los servicios definidos debe exponer un puerto HTTP mediante ports: o expose:. 
    mi-repo/
├── docker-compose.yml   ← aquí
├── backend/
└── frontend/
  2. Si no existe el docker-compose.yml, buscamos un Dockerfile en la raíz y generamos el compose automáticamente.
    Para esto necesitamos que diligencie el campo "Puerto del contenedor" en el formulario (el puerto que su aplicación escucha dentro del contenedor, por ejemplo 3000, 8080 u 80). 
    mi-repo/
├── Dockerfile           ← aquí
└── src/
  3. Si no encontramos ninguno de los dos en la raíz, el despliegue falla con un mensaje claro para que pueda corregirlo.

Archivo .env

  • Cárguelo en el campo correspondiente del formulario. No lo incluya en el repositorio.
  • Se escribe en la raíz del repositorio como .env antes de construir las imágenes.
  • Tamaño máximo: 64 KB.

Tokens y secretos

  • Este despliegue es público y temporal. No utilice claves de producción.
  • Use credenciales de prueba o entornos sandbox: Mercado Pago test, Stripe test keys, etc.
  • Si por error subió un token de producción, rótelo inmediatamente en el panel del proveedor.
  • El token de Git se cifra con AES-256-GCM antes de almacenarse. Use un token con permisos mínimos (solo lectura del repositorio).

Ciclo de vida

  • Construcción → READY → Activación (tras confirmar el pago) → 5 días activo.
  • Al expirar, el contenedor se destruye y el subdominio queda disponible nuevamente.
  • Puede mantener varios despliegues simultáneos, pero todos deben pertenecer al mismo proyecto.
